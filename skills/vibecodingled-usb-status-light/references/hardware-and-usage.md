# VibeCodingLed 硬件原理与使用说明

## 基本原理图

VibeCodingLed 使用 Digispark 风格 ATtiny85 USB 小板。客户使用时不需要外接线，直接插 USB 即可。

```text
电脑 USB-A/USB-C 口
  |
  |-- 5V ---------------------> 小板 5V / VCC
  |-- GND --------------------> 小板 GND
  |-- USB D- -----------------> ATtiny85 PB3 / P3  (V-USB 保留)
  |-- USB D+ -----------------> ATtiny85 PB4 / P4  (V-USB 保留)
                               |
                               |-- ATtiny85 固件
                               |     - usbPoll() 维持 USB HID 通信
                               |     - applyPayload() 解析电脑端状态命令
                               |     - updateOutputs() 根据状态刷新 LED 和蜂鸣器
                               |
                               |-- PB1 / P1 / LED_PIN=1
                               |     |
                               |     +-- 限流电阻 -- 板载 LED -- GND
                               |
                               |-- PB0 / P0 / BUZZER_PIN=0
                                     |
                                     +-- 有源蜂鸣器 -- GND
```

说明：

- `PB1/P1` 是状态 LED 控制脚。
- `PB0/P0` 是蜂鸣器控制脚，用作声音提醒。
- `PB3/P3`、`PB4/P4` 用作 USB 通信，不要再分配给 LED、蜂鸣器或其它外设。
- 固件采用 USB HID Feature Report，不需要把设备伪装成串口。
- LED 负责视觉提示，蜂鸣器只在需要客户注意的 `partial/done` 状态短鸣。
- 蜂鸣器默认 PWM duty 为 `8`。客户可让 AI 调整为更响或更轻，AI 不应要求客户手动编辑配置文件。

## USB 身份

| 阶段 | VID/PID | 含义 | 客户侧处理 |
| --- | --- | --- | --- |
| 运行态 | `16C0:05DF` | VibeCodingLed HID 固件正在运行 | 正常使用 |
| bootloader | `16D0:0753` | Micronucleus 固件上传窗口 | 等待或重新插拔；只有维护刷机时才处理驱动 |

客户收到的产品应默认处于运行态。如果只出现 bootloader，优先重新插拔并等待几秒。

## 从 0 到 1 使用说明

客户收到产品后按下面流程使用：

1. 从订单页、售后资料链接、随货二维码或 GitHub 下载完整的 `vibecodingled-usb-status-light` skill。GitHub 链接为 `https://github.com/lthirty/VibeCodingLed`；客户可点击 `Code` -> `Download ZIP`，解压后使用完整的 `skills/vibecodingled-usb-status-light` 文件夹。必须使用整个目录，不能只拿 `SKILL.md`。
2. 把 VibeCodingLed 插到电脑上能传数据的 USB 口。这个口应能插 U 盘、键盘或鼠标；只供电的充电口不行。
3. 打开 Codex、Cursor 或其它 AI 编程工具，让 AI 读取本 skill。只安装 Codex 也可以完成自动监控。
4. 对 AI 说：“请读取 VibeCodingLed USB 状态灯 skill，自动检测设备、安装依赖、配置 AI 编程状态监控，并验证 LED 灭、亮0.5秒灭0.5秒、亮1.5秒灭0.5秒、常亮四态，以及蜂鸣器提醒。”
5. AI 完成后，应向客户报告 USB 设备、HID 状态、监控源、当前聚合状态以及 LED/蜂鸣器效果。

客户判断是否正常工作：

| 检查项 | 正常结果 |
| --- | --- |
| USB 设备 | AI 报告识别到 `16C0:05DF 01AI VibeCodingLed` |
| USB 通信 | AI 报告 `error=ok` |
| 无 AI 编程任务 | LED 熄灭，蜂鸣器静音 |
| AI 任务全部运行中 | LED 亮 0.5 秒、灭 0.5 秒，蜂鸣器静音 |
| 部分线程停止且其它线程仍运行 | LED 亮 1.5 秒、灭 0.5 秒，蜂鸣器每 2 秒短鸣一次，共 3 次 |
| 所有已追踪线程停止 | LED 常亮，蜂鸣器每 4 秒三连短鸣一轮，共 3 轮 |
| Windows 后台监控 | 不应反复弹出黑色命令窗口 |

简单演示应验证 AI 任务状态联动，不要只做直接灯效控制。Codex 可按下面方式测试：

1. 新建第一个 Codex 对话，发送长任务：

```text
请写一份 Python 入门学习计划，至少 120 条，每条都要有具体练习任务。
```

2. 在第一个对话仍在生成时，新建第二个 Codex 对话，发送短任务：

```text
请列出 10 个整理电脑文件的建议。
```

3. 两个任务都在运行时 LED 应亮 0.5 秒、灭 0.5 秒且蜂鸣器静音；短任务先完成而长任务仍运行时 LED 应亮 1.5 秒、灭 0.5 秒且蜂鸣器短鸣 3 次；所有任务结束后 LED 应常亮且蜂鸣器三连短鸣 3 轮；客户确认后，如果没有其它后台 Codex 任务，LED 应熄灭且蜂鸣器静音。

客户看到常亮后，可以对 AI 说：

```text
我已经知道刚才完成的任务了，请清除已结束任务提醒，继续监控剩余任务。
```

AI 应执行确认动作，只清除已停止线程，不清除仍在运行的线程。若还有任务运行，LED 应回到亮 0.5 秒、灭 0.5 秒，蜂鸣器静音；后续其它任务停止时仍会再次亮 1.5 秒、灭 0.5 秒并短鸣。

也可以让 AI 自动组织这个测试：

```text
请根据 VibeCodingLed skill，判断当前 AI 工具是否能检测多个任务的开始和停止。如果可以，请帮我创建一个长任务和一个短任务，用它们验证：两个任务都运行时 LED 亮0.5秒灭0.5秒且蜂鸣器静音；一个任务停止、另一个任务仍运行时 LED 亮1.5秒灭0.5秒且蜂鸣器短鸣3次；所有任务结束后 LED 常亮且蜂鸣器三连短鸣3轮；我确认后 LED 熄灭且蜂鸣器静音。
```

如果当前本来就有其它 Codex 后台对话正在运行，最后一步可能不会熄灭，而是继续按剩余任务显示，这属于正常现象。

当前平台支持：

| 平台 / 工具 | 当前状态 |
| --- | --- |
| Windows + Codex | 已实现并实测；只安装 Codex 也能通过会话事件监控任务开始和停止 |
| Windows + Cursor | 已提供 Hook 安装脚本，重启 Cursor 后生效 |
| VS Code | USB 控制层可用；在 VS Code 中使用 Codex 时，状态来自 Codex，不是 VS Code 编辑器本身 |
| macOS / Linux | HID 原理上可用，但后台启动项和 AI 工具 Hook 需要按系统适配 |
| 其它 AI 工具 | 只要能在任务开始/停止时调用脚本，或能提供 active 线程快照，就可以接入 |

## LED 和蜂鸣器使用说明

| LED 效果 | 蜂鸣器效果 | 表示 |
| --- | --- | --- |
| 灯灭 | 静音 | 当前没有需要监控或确认的 AI 编程线程 |
| 亮 0.5 秒、灭 0.5 秒 | 静音 | 存在一个或多个 AI 编程线程，并且全部仍在运行 |
| 亮 1.5 秒、灭 0.5 秒 | 每 2 秒短鸣一次，共 3 次 | 至少一个线程已经停止，同时仍有其它线程在运行，需要查看 |
| 常亮 | 每 4 秒三连短鸣一轮，共 3 轮 | 所有已追踪线程都已经停止，需要查看最终结果 |

常亮不是一定代表失败，它代表“所有已追踪任务都结束，需要处理结果”。客户确认看过后，AI 执行 ACK，灯和蜂鸣器才会根据剩余任务回到运行态或熄灭/静音。

常亮是一个可确认的提醒，不是永久锁死状态。产品本体没有确认按钮，确认入口是 AI 工具和 skill。客户确认看过结果后，AI 应执行确认动作，清除已停止提醒，并继续监控剩余运行线程。

## 蜂鸣器音量

默认音量为 PWM duty `8`，设置保存在 `%USERPROFILE%\.vibecodingled\settings.json`。AI 工具根据客户自然语言执行：

| 客户说法 | 命令 |
| --- | --- |
| 声音响一点 | `python scripts\vibe_led.py buzzer louder` |
| 声音轻一点 | `python scripts\vibe_led.py buzzer quieter` |
| 恢复默认音量 | `python scripts\vibe_led.py buzzer default` |
| 当前音量是多少 | `python scripts\vibe_led.py buzzer get` |

档位顺序为 `2/4/8/16/24/32/48/64/96/128`。默认 `8` 已通过样机验证；如果环境噪声较大，可逐步调到 `16` 或 `24`。调整后，后续 Codex/Cursor 监控下发状态时会自动带上新的 duty。

## 客户说法示例

客户可以对 AI 工具说：

```text
读取 VibeCodingLed USB 状态灯 skill，帮我自动安装并监控 Codex 的 AI 编程状态。
```

或者：

```text
我已经把状态灯插到 USB 口了，帮我配置成：没有需要监控或确认的任务时灯灭且蜂鸣器静音；所有任务运行时亮0.5秒灭0.5秒且蜂鸣器静音；部分任务完成但仍有任务运行时亮1.5秒灭0.5秒且蜂鸣器短鸣3次；所有任务完成时常亮且蜂鸣器三连短鸣3轮。
```

AI 工具应自动完成：

1. 检测 USB HID 设备。
2. 安装 Python HID 运行依赖。
3. 验证 LED 四态和蜂鸣器提醒。
4. 配置当前 AI 工具的 Hook、自动化或快照同步。
5. 告诉客户当前是否已经能自动监控。

## 故障排查

| 现象 | 优先检查 |
| --- | --- |
| `list` 找不到设备 | 换 USB 口，避免无源集线器，检查设备管理器 VID/PID |
| 只看到 `16D0:0753` | 设备停在 bootloader，重新插拔等待运行态 |
| Windows 错误码 28 | 通常是 bootloader 驱动缺失；客户正常使用不应停在这里 |
| 能识别但 LED 不亮 | 确认固件 LED 引脚是否为 `PB1/P1`，再测试 `set done` |
| 蜂鸣器不响 | 确认蜂鸣器是否接在 `PB0/P0`，再测试 `set partial` 或 `set done` |
| 蜂鸣器太响或太轻 | 让 AI 执行 `buzzer quieter` 或 `buzzer louder`，默认 duty 为 `8` |
| LED/蜂鸣器与 AI 状态不一致 | 检查监控源是否真的在调用 `thread sync`，查看 `%USERPROFILE%\.vibecodingled\threads.json` |

## 交付边界

- 客户交付包应包含实物产品和本 skill。
- 产品应预刷运行态固件，客户不应被要求安装 PlatformIO 或刷机。
- AI 工具读取 skill 后应能完成安装、检测和配置。
- 如果某个 AI 工具不提供 Hook、自动化或活跃线程查询能力，skill 必须如实提示无法完全自动监控，并提供手动或半自动方案。
